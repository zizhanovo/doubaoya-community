# 小红书评论分析（都爆鸭）

嘎！把一条小红书笔记的评论区端回来给你看。给本鸭一个笔记链接或 noteId，本鸭就把这条笔记的一级评论捞出来，铺成「评论/点赞」表格，再顺手帮你把评论区的情绪、需求、竞品声音复盘一遍。评论多就靠游标一页页翻。

> 数据走 **doubaoya.com** 一条线，鉴权用你自己的口令（环境变量 `DOUBAOYA_API_KEY`，形如 `dyh_…`）。

---

## 适用场景

| 场景 | 怎么用 | 拿到什么 |
|------|--------|----------|
| **看评论区反馈** | 贴笔记链接，拉出评论区一页页看 | 真实用户在说什么 |
| **舆情/口碑复盘** | 拉完评论做四维情感分析 | 积极/负面/需求/竞品一眼清 |
| **挖用户需求** | 从评论里找"求链接""求教程"等诉求 | 给下一篇选题/产品找方向 |
| **竞品声音监控** | 看评论里有没有点名竞品 | 摸清对手在用户心里的位置 |
| **爆款评论拆解** | 对爆款笔记拉评论，看高赞评论为什么火 | 学互动话术、抄金句 |

## 工作流（4 步）

### 1. 提取 noteId
从用户给的内容里拿到 **noteId**：
- 用户给链接（如 `https://www.xiaohongshu.com/explore/6a2ac3020000000035022d8e`）→ 从链接里提取末段的 noteId。
- 用户直接给 noteId（如 `6a2ac3020000000035022d8e`）→ 直接用。
- 没给 → 主动问一句「请把小红书笔记链接发给本鸭」。
- 上一轮已查过某条笔记、本轮只说「下一页」「再翻」之类 → 沿用上一轮的 noteId。

### 2. 调用脚本取数
首页（不传 `--cursor`，首页游标留空即可）：
```bash
python3 "$SKILL_PATH/scripts/fetch_xhs_comments.py" "<noteId>"
```
翻页（把上一页 `data` 里的 `nextCursor` 原样传进来）：
```bash
python3 "$SKILL_PATH/scripts/fetch_xhs_comments.py" "<noteId>" --cursor "<上一页nextCursor>"
```
脚本把成功信封里的 `data` 以 JSON 打到 stdout。**每次只跑一次脚本拉一页**，别在同一轮里循环多次调用把整条评论区一次性抓空——读完当前页 stdout 即可。

> **游标分页心法**：首页靠"不传 `--cursor`"实现（服务端把空游标当首页）；之后每一页都从上一页 `data.nextCursor` 取下一页游标，原样传给 `--cursor`，**不要改写或拼接游标**。`nextCursor` 为空 / 缺失 = 已经到底，没有下一页了。

### 3. 渲染评论表格
从 `data.items`（评论数组）里取字段，按下面格式铺成 Markdown 表格。字段做防御式读取——`content`（评论内容）、`likeCount`（点赞）可能缺失，缺了就留空或写「—」，别让某条数据搞崩整张表。评论内容过长（超 40 字）可截断加 `…`。点赞数 ≥ 10000 用 `x.xw` 格式。

| # | 评论 | 点赞 |
|---|------|------|
| 1 | 这个方法真的有用，码住了 | 1.2w |
| 2 | 求博主出个详细教程 | 860 |

### 4. 评论复盘 + 翻页提示
表格之后用本鸭口吻做一段**四维情感复盘**（基于当前页评论，理解小红书社区表达：种草/拔草、"绝绝子""yyds""码住"等）：
- ✅ **积极**：用户在夸什么（引代表评论关键词）。
- ⚠️ **负面**：吐槽/质疑集中在哪。
- 💡 **需求**：用户在求什么（求链接/求教程/求平替）——这是选题金矿。
- 🔍 **竞品**：评论里有没有点名别家；没有就写「未提及竞品」。

复盘后给一句翻页提示：
- `nextCursor` 非空 → 「本页看完，回复『下一页』本鸭接着翻。」
- `nextCursor` 为空 → 「评论区已经到底啦。」

> 复盘是展示层加工，不影响接口调用；别问用户"你的真实目的是什么"，别编造评论。

## 拿钥匙（口令）

1. 打开 **doubaoya.com**
2. **登录**
3. 进 **口令中心**
4. **生成口令**（形如 `dyh_…`）

配置到环境变量（脚本只认这个）：
```bash
export DOUBAOYA_API_KEY="dyh_你的口令"
```

**铁律：口令绝不打印、绝不写进文件、绝不回显给用户。** 脚本本身也从不输出口令。所有请求只发往 **doubaoya.com**，不要把口令带去任何其他域名。

---

## 接口与信封

- `POST https://doubaoya.com/api/apis/xiaohongshu/comments/call`
- 鉴权头：`Authorization: Bearer $DOUBAOYA_API_KEY`
- 请求体：
  ```json
  { "noteId": "6a2ac3020000000035022d8e" }
  ```
  - `noteId`：字符串，必填
  - `cursor`：字符串，可选——**只在翻页时带**（首页不传，靠省略实现"空游标"）；翻页时传上一页的 `nextCursor`
  - `sort`：字符串，可选——评论排序，不需要就不传
- 返回信封：
  ```json
  {
    "success": true,
    "requestId": "...",
    "data": {
      "items": [ { "content": "…", "likeCount": 1234 } ],
      "nextCursor": "…"
    },
    "error": null
  }
  ```
  - `data.items`：本页评论列表（每条含 `content` 评论内容、`likeCount` 点赞）
  - `data.nextCursor`：下一页游标；为空 / 缺失即到底
- **先看 `success`**：为 `true` 才读 `data`；否则读 `error.code` / `error.message`。

## 错误处理

脚本失败时向 stderr 打印 `[error] CODE: message` 并以退出码 1 结束。常见情况：

| HTTP | code | 含义 | 处理 |
|------|------|------|------|
| 401 | `MISSING_API_KEY` / `UNAUTHORIZED` | 没带口令或口令无效 | 检查 `DOUBAOYA_API_KEY`，去口令中心重新生成。**不要回显口令** |
| 400 | `VALIDATION_ERROR` | 参数不合法（如 noteId 为空 / 链接无效） | 核对笔记链接，重新提取 noteId |
| 402 | `INSUFFICIENT_CREDITS` | 额度不足 | 去 doubaoya.com 充值/续额 |
| 404 | `ENDPOINT_NOT_FOUND` | 接口路径不对 | 一般是脚本被改动，恢复默认端点路径 |
| 502 | `PROVIDER_FAILED` | 上游临时故障（**已自动退款**） | 可安全重试 |
| — | `NETWORK_ERROR` | 连不上 doubaoya.com | 检查网络后重试 |

> `502 PROVIDER_FAILED` 会自动退款，重试是安全的，不会重复扣费。拉到 0 条评论时，提示「该笔记暂无评论」并建议核对笔记是否存在/已删除。
